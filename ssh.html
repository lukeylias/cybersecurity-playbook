SSH (Secure Shell)

Secure Remote Access & Authentication

SSH gives you a secure, encrypted connection to a remote machine's command line. It's how you manage servers, transfer files, and tunnel traffic ‚Äî all over an untrusted network. It runs on port 22 by default.

üîê How SSH Works
SSH creates an encrypted channel between your machine (the client) and a remote machine (the server). Everything sent over this channel ‚Äî commands, passwords, file contents ‚Äî is encrypted, unlike older protocols like Telnet which send everything in plain text.
üí° SSH vs Telnet

Telnet does the same basic job as SSH ‚Äî remote command line access ‚Äî but sends everything unencrypted, including your password. Telnet should never be used on real networks. SSH replaced it entirely.

Basic Connection
ssh user@192.168.1.10
Connect to a remote machine by IP address as a specific user
ssh user@hostname
Connect using a hostname instead of IP (requires DNS resolution)
ssh -p 2222 user@192.168.1.10
Connect on a non-standard port (default is 22)
$ ssh analyst@10.10.45.12
The authenticity of host '10.10.45.12' can't be established.
ED25519 key fingerprint is SHA256:xR4kL...mN9pQ.
Are you sure you want to continue connecting (yes/no/[fingerprint])? yes
Warning: Permanently added '10.10.45.12' (ED25519) to the list of known hosts.
analyst@10.10.45.12's password:
analyst@target:~$
üí° First-time fingerprint

The first time you connect to a server, SSH shows you its fingerprint and asks you to verify. This protects against man-in-the-middle attacks ‚Äî if the fingerprint changes on a future connection, SSH will warn you that something may be wrong. The fingerprint gets saved to ~/.ssh/known_hosts.

üîë Authentication Methods
SSH supports two main ways to prove you are who you say you are: passwords and SSH keys. Keys are more secure and widely preferred.
Password Authentication
ssh user@192.168.1.10
The server prompts for the user's password. Simple but less secure ‚Äî passwords can be brute-forced, guessed, or intercepted if reused.
Key-Based Authentication
ssh-keygen
Generates a key pair: a private key (stays on your machine, never shared) and a public key (goes on the server)
ssh-keygen -t ed25519
Generates a key pair using the Ed25519 algorithm (modern and recommended)
ssh-copy-id user@192.168.1.10
Copies your public key to the remote server's authorized_keys file
ssh -i ~/.ssh/my_key user@192.168.1.10
Connect using a specific private key file
$ ssh-keygen -t ed25519
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
Enter passphrase (empty for no passphrase):
Your identification has been saved in /home/user/.ssh/id_ed25519
Your public key has been saved in /home/user/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:aBcDe...xYz12 user@machine
üí° How key auth works

Your private key stays on your machine and should never be shared ‚Äî it's like a physical key. The public key goes on any server you want to access ‚Äî it's like a lock. When you connect, SSH uses a cryptographic challenge: the server uses your public key to create a puzzle only your private key can solve. No password ever crosses the network.

‚ö†Ô∏è Protect your private key

If someone gets your private key, they can access every server it's authorised on. Always set a passphrase when generating keys, and set file permissions to chmod 600 ~/.ssh/id_ed25519 ‚Äî SSH will refuse to use a private key with loose permissions.

üìÅ File Transfer Over SSH
SSH isn't just for remote terminals ‚Äî it's also the secure transport layer for copying files between machines. Two main tools: scp for simple copies and sftp for interactive file browsing.
SCP ‚Äî Secure Copy
scp file.txt user@192.168.1.10:/home/user/
Copy a local file to a remote machine
scp user@192.168.1.10:/var/log/auth.log ./
Copy a remote file to your local machine
scp -r projects/ user@192.168.1.10:/home/user/
Copy an entire directory recursively
scp -P 2222 file.txt user@192.168.1.10:/tmp/
Copy over a non-standard SSH port (note: uppercase -P for scp, lowercase -p for ssh)
SFTP ‚Äî Secure File Transfer
sftp user@192.168.1.10
Opens an interactive file transfer session
ls / lls
List remote files / list local files (within sftp session)
get remotefile.txt
Download a file from the remote machine
put localfile.txt
Upload a file to the remote machine
exit
Close the sftp session
üõ°Ô∏è SSH for Pentesting
SSH shows up constantly in penetration testing ‚Äî as an attack surface, as a tool for maintaining access, and as a way to tunnel traffic through compromised machines.
Common Pentest Uses
ssh -L 8080:localhost:80 user@target
Local port forwarding: access the target's port 80 through your local port 8080
ssh -D 9050 user@target
Dynamic port forwarding: creates a SOCKS proxy through the target (useful for routing tools through a compromised host)
ssh -N -f -L 3306:db-server:3306 user@target
Background tunnel to access an internal database through the target
üí° Why this matters

In pentesting, you often compromise a machine that has access to internal networks you can't reach directly. SSH tunnelling lets you pivot ‚Äî routing your traffic through the compromised host to reach those internal systems. This is one of the most practical skills in the Jr Penetration Tester path.

‚ö†Ô∏è Weak SSH configurations to look for

During security assessments, check for: password authentication still enabled (should use keys only), root login permitted (PermitRootLogin yes in sshd_config), default port 22 (makes automated scanning easier), and outdated SSH versions with known vulnerabilities.
