Packages & Logs

Software Management & System Monitoring

Two essential Linux skills: installing and managing software through package managers, and reading system logs to understand what's happening on your machine.

üì¶ Package Management
A package manager handles installing, updating, and removing software. Instead of downloading files from websites, you use a single tool that pulls from trusted repositories, handles dependencies automatically, and keeps everything up to date.
Which package manager?

On Debian/Ubuntu systems (the most common), you'll use apt. On Red Hat/Fedora/CentOS, it's dnf (or the older yum). The concepts are the same ‚Äî only the command names change.

apt ‚Äî Debian / Ubuntu
sudo apt update
Refreshes the list of available packages from repositories. Always run this before installing anything to make sure you're pulling the latest versions.
sudo apt upgrade
Upgrades all installed packages to their newest versions. Combine with update: sudo apt update && sudo apt upgrade
sudo apt install nginx
Installs a package and any dependencies it needs. Apt figures out what else is required and pulls it all in.
sudo apt remove nginx
Removes the package but keeps its config files, in case you reinstall later.
sudo apt purge nginx
Removes the package and its config files ‚Äî a clean uninstall.
apt search text-editor
Searches available packages by keyword. Doesn't need sudo since it's read-only.
apt list --installed
Shows all currently installed packages.
$ sudo apt install nginx
Reading package lists... Done
Building dependency tree... Done
The following NEW packages will be installed:
  nginx nginx-common nginx-core
0 upgraded, 3 newly installed, 0 to remove.
Do you want to continue? [Y/n] Y
dnf ‚Äî Red Hat / Fedora / CentOS
sudo dnf install nginx
Installs a package. dnf handles dependency resolution automatically, same as apt.
sudo dnf remove nginx
Removes a package and any dependencies that are no longer needed.
sudo dnf update
Updates all installed packages. In dnf, update and upgrade do the same thing.
dnf search text-editor
Searches for packages by keyword across all available repositories.
dnf list installed
Lists all installed packages on the system.
‚ö†Ô∏è Why sudo?

Installing and removing software changes the system, so it requires root privileges. That's why most package commands start with sudo. Search and list commands are read-only, so they don't need it.

üìã Viewing Logs
Linux logs everything ‚Äî logins, errors, services starting and stopping, network activity. Logs are how you figure out what went wrong, spot suspicious activity, or just check that things are running normally. Most logs live in /var/log/ or are managed by systemd's journal.
Key Log Files
FileWhat it contains
/var/log/syslogGeneral system activity ‚Äî the catch-all log. Services, kernel messages, and anything not captured by a more specific log.
/var/log/auth.logAuthentication events ‚Äî login attempts, sudo usage, SSH sessions. First place to check for unauthorized access.
/var/log/kern.logKernel messages ‚Äî hardware issues, driver errors, boot problems.
/var/log/dpkg.logPackage manager history ‚Äî what was installed, removed, or upgraded and when.
/var/log/apache2/Web server logs (if Apache is installed). Contains access.log and error.log.
journalctl ‚Äî Systemd Journal
Modern Linux systems use systemd, which has its own logging system called the journal. The journalctl command is the main way to query it.
journalctl
Shows the full system journal, oldest first. Can be very long ‚Äî usually you'll want to filter it.
journalctl -e
Jumps to the end of the journal ‚Äî the most recent entries. Usually what you want.
journalctl -f
Follows the log in real time, showing new entries as they appear. Like a live feed. Press Ctrl+C to stop.
journalctl -u nginx
Shows logs for a specific service only. Great for debugging why a service won't start.
journalctl -u ssh --since "1 hour ago"
Filters by service and time. You can also use dates like "2025-01-15" or "yesterday".
journalctl -p err
Shows only error-level messages. Priority levels: emerg, alert, crit, err, warning, notice, info, debug.
journalctl -b
Shows logs from the current boot only. Use -b -1 for the previous boot.
Reading Log Files Directly
cat /var/log/auth.log
Dumps the entire file. Fine for small logs, overwhelming for large ones.
tail -n 20 /var/log/syslog
Shows the last 20 lines. Tail is your go-to for checking recent log entries quickly.
tail -f /var/log/syslog
Follows the file in real time ‚Äî new lines appear as they're written. Like journalctl -f but for a specific file.
head -n 10 /var/log/dpkg.log
Shows the first 10 lines. Useful for checking the start of a log or seeing the format.
grep "Failed password" /var/log/auth.log
Searches for a specific pattern in a log file. This example finds all failed login attempts.
grep -i "error" /var/log/syslog | tail -n 20
Combining tools: find all errors (case-insensitive) then show only the 20 most recent.
$ grep "Failed password" /var/log/auth.log | tail -n 3
Jan 28 03:14:22 server sshd[4821]: Failed password for root from 192.168.1.105 port 52341 ssh2
Jan 28 03:14:25 server sshd[4821]: Failed password for root from 192.168.1.105 port 52341 ssh2
Jan 28 03:14:28 server sshd[4823]: Failed password for invalid user admin from 10.0.0.42 port 39102 ssh2
üí° Reading log output

Most log lines follow the same format: timestamp hostname service[pid]: message. Once you recognize this pattern, scanning logs becomes much faster. Look for the timestamp to find when something happened, and the message to understand what.

‚öôÔ∏è Services & Boot
Services are background programs that keep your system running ‚Äî web servers, SSH, databases. The systemctl command controls these services and determines which ones start automatically when the system boots. The very first process (PID 0) starts at boot and spawns everything else.
systemctl ‚Äî Service Management
systemctl start nginx
Starts a service immediately for this session
systemctl stop nginx
Stops a running service
systemctl restart nginx
Stops and restarts a service (useful after config changes)
systemctl status nginx
Shows if a service is running, its PID, and recent log output
systemctl enable nginx
Enables a service to start automatically at boot
systemctl disable nginx
Prevents a service from starting at boot
$ systemctl status nginx

üí° Enable vs Start

start runs a service right now but it won't survive a reboot. enable hooks it into the boot process so it launches automatically every time the system starts. To do both at once: systemctl enable --now nginx.

‚è∞ Scheduled Tasks (Cron)
Cron is Linux's built-in task scheduler. It runs commands automatically on a schedule ‚Äî backups at midnight, log cleanup every Sunday, health checks every 5 minutes. Each user has their own crontab file that holds their scheduled jobs.
Managing Cron
crontab -e
Edit your crontab file (opens in a text editor)
crontab -l
List all your scheduled cron jobs
crontab -r
Remove all your cron jobs (be careful with this)
sudo crontab -u alice -l
View another user's crontab
Cron Syntax
Each cron job follows a five-field time format that specifies exactly when the command should run.

‚ö†Ô∏è Cron gotchas

Cron jobs run with a minimal environment ‚Äî commands that work in your terminal might fail in cron because PATH is different. Use full paths to commands (like /usr/bin/python3 instead of python3) and redirect output to a log file so you can debug failures.
